# Stage 04 — Pull request

File the PR. Run the Cursor BugBot review loop. Iterate until clean. Merge.

---

## Pre-PR checks

### 1. Type-check + lint one more time

We routinely catch a stale type error or unused import right before opening a PR. Save yourself the round-trip:

```bash
npx tsc --noEmit
```

If biome / prettier are configured, run them too.

### 2. Diff size sanity check

```bash
git diff main --stat
```

Review the file count. If the diff includes significantly more files than you actually changed (e.g. hundreds of regenerated files for a 2-file change), create a clean branch from `main` with only the necessary changes. Common causes of bloat:

- Full `firebase-helpers/` regeneration when only a few fields are needed → manually edit the generated type instead
- Unrelated formatting changes from editor or generator runs
- Accidental commits from another concurrent branch

### 3. Cross-repo plugin mirror (if applicable)

If the PR touches `scripts/openit-plugin/`, you also need to land the same change in `/web` so production users get it. **The PR can be opened first; the `/web` mirror lands at merge time.** But plan ahead:

1. Identify every file under `scripts/openit-plugin/` your PR changes.
2. Note the corresponding `/web` paths: `web/packages/app/public/openit-plugin/...`.
3. Bump version in `web/packages/app/public/openit-plugin/manifest.json` when you mirror.

The plan from stage 02 should already list these as checklist items. This is the moment to execute them.

## Create the PR

```bash
gh pr create --draft --base main --title "<type>(PIN-XXXX): <Description>" --body "$(cat <<'EOF'
## Summary

<1–3 bullets of what changed and why>

## Test plan

- [x] vitest — N passing
- [x] cargo test — clean
- [x] tsc --noEmit — clean
- [x] Manual scenario 1 — <result>
- [x] Manual scenario 2 — <result>

## Related

- Linear: PIN-XXXX
- Plan: \`auto-dev/plans/YYYY-MM-DD-PIN-XXXX-...md\`

🤖 Generated with [Claude Code](https://claude.com/claude-code)
EOF
)"
```

### Naming convention (Conventional Commits)

Title format: `<type>(PIN-XXXX): <Description>`

- First letter of description **MUST be uppercase**
- Common types: `feat`, `fix`, `perf`, `refactor`, `chore`, `docs`, `test`, `style`
- Scope is the Linear ticket — always include it

Examples:
- `feat(PIN-5775): Version-gated plugin overlay`
- `fix(PIN-5775): Datastore deletions counted as pending`
- `perf(PIN-5775): Skip pre-pull when no local changes`

### Open as DRAFT first

Always open as `--draft`. Mark Ready for Review only after BugBot is clean and CI is green.

## Updating an existing PR description

**Never blindly overwrite an existing PR description.** `gh pr edit --body ...` replaces the whole body — if you regenerate it from scratch, you destroy content that did not come from you.

Preserve at minimum:

1. **The Cursor summary block** — everything between `<!-- CURSOR_SUMMARY -->` and `<!-- /CURSOR_SUMMARY -->`. Cursor writes and re-writes this automatically; overwriting it loses the latest risk assessment and severs the BugBot linkage.
2. **Any screenshots, GIFs, or videos the human added** — `![...](...)`, `<img ...>`, GitHub user-images URLs, asset URLs.
3. **Human-written sections** that were not in your original template (e.g. "Manual testing", "Deploy notes").
4. **Reviewer conversation context** — anything like "Addresses feedback from @alice re: caching."

### Required workflow for an edit

```bash
gh pr view $PR_NUMBER --json body --jq '.body' > /tmp/pr-$PR_NUMBER-body.md
# edit /tmp/pr-$PR_NUMBER-body.md in place — touch only sections YOU originally wrote
gh pr edit $PR_NUMBER --body-file /tmp/pr-$PR_NUMBER-body.md
```

If you can't confidently identify which parts are yours vs. someone else's, **ask the user** before pushing.

## BugBot loop

OpenIT runs Cursor BugBot. **GO IN A LOOP. DO NOT STOP UNTIL GREEN.** Either fix every finding or reply with rationale on every finding — no silent skips.

### 1. Trigger

> **The trigger is exactly `@cursor review`** on a line by itself. Variants like `@cursor please re-review`, `@cursor take another look`, `/bugbot`, or just `@cursor` with no verb are **no-ops**. Always include the literal `@cursor review` line.

First, **`git pull`** — BugBot can auto-push small mechanical fixes (variable shadowing, dead-code, trivial typos) directly to your branch. If you don't pull before iterating, your next push silently reverts BugBot's fix on rebase.

Second, check if BugBot is already running. Re-triggering while a review is in progress restarts it from scratch, wasting time:

```bash
PR_NUMBER=<your PR number>
HEAD_SHA=$(gh pr view $PR_NUMBER --json headRefOid --jq '.headRefOid')

gh api repos/{owner}/{repo}/commits/$HEAD_SHA/check-runs \
  --jq '.check_runs[] | select(.name == "Cursor Bugbot") | {status, started_at}'
```

If `status` is `"in_progress"`, **don't re-trigger** — wait. Otherwise:

```bash
gh pr comment $PR_NUMBER --body "@cursor review"
```

#### Default: open-ended

Most of the time, just `@cursor review` (optionally with a brief "what changed since last review" line). The whole point of BugBot is catching issues *you* didn't think of — listing your concerns up-front anchors the review on what you're already worried about and tends to suppress the unknown-unknowns that are the real value-add.

```
@cursor review

Latest 5 commits since 15:36 — substantial batch covering schemas,
agent template, and the bundled-fallback path.
```

#### Focusing the review (use sparingly)

For broad diffs or non-obvious design choices, append specific concerns as bullets after the mention. BugBot reads them and addresses each one explicitly.

```
@cursor review

Latest 5 commits since 15:36 — substantial batch.

Specifically vet:
- The Rust `#[serde(default, skip_serializing_if = "Option::is_none")]`
  on the new field — backwards-compat with existing manifests?
- Force-push semantics in sync-resolve-conflict.mjs (`pulled_at_mtime_ms = 1`
  sentinel) — any race with concurrent local edits?
- `pushInFlightByRepo` Set vs a per-(repo, prefix) lock model — overkill
  or right-sized?
```

What works:
- **Bullets, not prose.** Each concern gets answered separately.
- **Name the commit/SHA range.** Otherwise BugBot tends to rescan the whole PR.
- **Phrase as questions.** Yes/no or "race?" gets a directly comparable answer.
- **Three to five concerns max.** More dilutes; fewer is fine.

When NOT to focus: a fresh review with no specific worry, or a small one-finding fix. Plain `@cursor review` is faster and BugBot's defaults are good.

> **Status updates and fix summaries belong in per-finding replies, not the trigger comment.** The trigger is for what the reviewer should look at, not what you've been doing.

### 2. Wait

BugBot typically takes 3–7 minutes. **Actively poll. Don't sleep for a fixed time.**

```bash
gh api repos/{owner}/{repo}/commits/$HEAD_SHA/check-runs \
  --jq '.check_runs[] | select(.name == "Cursor Bugbot") | {status, conclusion, started_at, completed_at}'
```

Run, sleep ≥45 s, run again. Repeat until `status` is `"completed"`.

### 3. Read findings

> **CRITICAL: BugBot posts in TWO places — check BOTH.**
>
> 1. **Review summary** — posted as a PR review (issue count: "found N potential issues" or "found no new issues")
> 2. **Inline findings** — posted as PR review comments on specific diff lines (the actual bug descriptions)
>
> **`gh pr view --comments` only shows issue-level comments, NOT review comments.** If you only check that, you'll miss BugBot findings entirely. Use the API endpoints below.

Check the review summary:

```bash
gh api repos/{owner}/{repo}/pulls/$PR_NUMBER/reviews \
  --jq '[.[] | select(.user.login == "cursor[bot]")] | last | .body'
```

If it says `"found no new issues"` and CI is green → done. Otherwise read the inline findings:

```bash
# Method 1: REST pull request comments
gh api repos/{owner}/{repo}/pulls/$PR_NUMBER/comments \
  --jq '.[] | select(.user.login == "cursor[bot]" or .user.login == "cursor") | {id, path, line, body}'

# Method 2: GraphQL review threads (newer BugBot runs)
gh api graphql -f query='
query {
  repository(owner: "{owner}", name: "{repo}") {
    pullRequest(number: '$PR_NUMBER') {
      reviewThreads(last: 30) {
        nodes {
          id
          isResolved
          comments(first: 1) {
            nodes { databaseId body createdAt author { login } path line }
          }
        }
      }
    }
  }
}' --jq '.data.repository.pullRequest.reviewThreads.nodes[]
  | select(.comments.nodes[0].author.login == "cursor")
  | {threadId: .id, path: .comments.nodes[0].path, line: .comments.nodes[0].line, body: .comments.nodes[0].body}'
```

Why both? BugBot sometimes posts as PR comments (REST), sometimes as review thread comments (GraphQL only). Checking only one is how you incorrectly classify an iteration as "clean."

Each finding has: file path, line number, severity (Critical / High / Medium / Low), description, optional "Additional Locations."

### 4. Triage

**Check the original plan first.** Before fixing, re-read stage 02's implementation plan. Verify each proposed fix doesn't contradict the plan's design decisions or architectural choices. BugBot doesn't know about the plan — it may flag intentional trade-offs.

**BugBot frequently produces false positives.** Don't blindly fix every finding. Read the actual code, understand the execution flow.

Triage each finding:

- **Fix** — Real bug or correctness issue that would cause incorrect behavior at runtime
- **Fix** — Real backwards-compat or runtime issue
- **Won't fix** — False positive (theoretical issue that can't happen, intentional behavior)
- **Won't fix** — Cosmetic (style, naming, duplicate labels) and doesn't affect correctness
- **Won't fix** — Preference / pattern we don't care about
- **Ask the engineer** — When uncertain whether a finding matters

### 5. Respond — every finding gets a reply

**MANDATORY.** No finding gets left without a response — fix or skip. If a finding has no reply, there's no record of what happened.

For **fixed** findings — reply with the commit SHA, then resolve the thread:

```bash
gh api repos/{owner}/{repo}/pulls/$PR_NUMBER/comments/{comment_id}/replies \
  -f body="Fixed in <commit SHA> — <brief description>"

gh api graphql -f query='mutation {
  resolveReviewThread(input: {threadId: "<thread_id>"}) {
    thread { isResolved }
  }
}'
```

For **won't fix** — reply with rationale, do NOT resolve (leave it open for the engineer):

```bash
gh api repos/{owner}/{repo}/pulls/$PR_NUMBER/comments/{comment_id}/replies \
  -f body="Skipping — <reason>"
```

### 6. Fix

**One fix per commit, named after the BugBot finding title.** This gives 1:1 traceability between findings and commits.

For each finding that needs a fix:

0. **`git pull`** — confirm BugBot didn't auto-push a fix while you were planning.
1. Read the affected file(s) at the flagged line(s)
2. Understand the root cause — don't just patch the symptom
3. Fix
4. **Run lint and type-check before committing:**
   ```bash
   npx tsc --noEmit
   ```
   Fix any errors before proceeding. Don't commit code that doesn't compile.
5. Commit using `Re:` followed by the **exact BugBot finding title**

```bash
# BugBot title: "Stale regex still matches removed pf.mcp.callTool pattern"
git commit -m "Re: Stale regex still matches removed pf.mcp.callTool pattern"
```

Do NOT batch multiple BugBot fixes into one commit. If BugBot raises 3 findings and you fix all 3, that's 3 separate commits.

### 7. Update the plan

After each iteration, append a **BugBot Review Log** to the bottom of the plan file (`auto-dev/plans/<plan-file>.md`):

```markdown
## BugBot Review Log

### Iteration 1 (YYYY-MM-DD)

| # | Finding | Severity | Disposition | Commit / Reason |
|---|---------|----------|-------------|-----------------|
| 1 | Synchronous loop after await blocks processing | High | Fixed | `abc1234` |
| 2 | Preview limit exceeds artifact threshold | Low | Skipped | Intentional — 500-char preview is for agent context, not user display |

### Iteration 2 (YYYY-MM-DD)

| # | Finding | Severity | Disposition | Commit / Reason |
|---|---------|----------|-------------|-----------------|
| 1 | (none — clean run) | — | — | — |
```

### 8. Re-trigger

Type-check one more time across the full batch:

```bash
npx tsc --noEmit
```

Then check BugBot isn't still running, then trigger:

```bash
gh pr comment $PR_NUMBER --body "@cursor review"
```

### 9. Loop until clean

**Exit condition: clean BugBot run AND all CI checks passing.** Every batch of fixes (or "won't fix" replies) gets a fresh BugBot trigger.

The loop:
1. Trigger
2. Wait
3. Read findings
4. Triage against plan (fix / won't fix)
5. For fixes: changes + one commit per finding using exact title
6. For won't fix: reply with rationale (don't resolve)
7. Update the plan's BugBot Review Log
8. Trigger again — mandatory even if all findings were "won't fix"
9. Repeat from step 2

Typically 2–5 iterations.

**At 3 iterations without a clean run, pause and self-audit.** Review your entire diff:
- Every HTTP call has a status check before reading the body
- Every adapter or service path has consistent headers
- Every caller of a modified function/struct has been updated
- Every error path has a meaningful user-facing message
- Every new dependency call has nil/null safety matching existing patterns

**At 10 iterations, stop and analyze.** Are the same areas getting flagged repeatedly? Are fixes for one finding introducing new findings? The design may need rethinking, not more patches. Write up the analysis in the BugBot log and present it to the engineer.

### Stop condition

Keep iterating until the only remaining findings are **Low** severity. At that point reply on each Low thread with a rationale (or fix), resolve, and move to merge.

Watch for:
- **BugBot auto-pushes** — it can land trivial mechanical fixes itself. **Always `git pull` before iterating.**
- **Pre-existing CI red on `main`** — if your PR's CI fails for the same reasons (auto-generated firebase-helpers, cargo fmt), that's not a regression you introduced.
- **Real issues vs. style nits** — BugBot is good at structural correctness; less good at project-specific conventions. Push back on style nits that conflict with patterns in `auto-dev/00-autodev-overview.md` or `CLAUDE.md`.

---

## After clean: merge

1. Mark the PR Ready for Review (un-draft).
2. **If the change touched `scripts/openit-plugin/`**, execute the cross-repo mirror:
   - Copy each changed script into `/web/packages/app/public/openit-plugin/scripts/<file>` (mirroring path).
   - Bump `web/packages/app/public/openit-plugin/manifest.json` version.
   - Open + merge a `/web` PR.
   - Verify: any reconnect now pulls the new script down to `~/OpenIT/<orgId>/.claude/scripts/`.
3. Merge the openit-app PR.
4. Add a final comment to the Linear ticket with the PR URL + a one-line summary.
