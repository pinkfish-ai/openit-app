## Problem

Phase 1 made **filestore** local-first: the bound folder stays at `~/OpenIT/local/`, the engine filters remote to `openit-*`, each collection gets its own manifest entry, and every `openit-<name>` collection round-trips with `filestores/<name>/`.

Every other content engine still runs the V1 model — remote-as-source-of-truth, single-default discovery, shared manifest. Concretely, the **knowledge-base engine** today:

- Discovers a single hardcoded default KB by name and clobbers anything else
- Has no `openit-*` prefix filter, so a user's unrelated KBs on the cloud either get pulled into the OpenIT folder or fight for the same default slot
- Stores all KB state in a single shared manifest (same architecture flaw the filestore architecture-fix addressed)
- Has no concept of per-KB folder routing — local-only KBs created by the user can't push up
- Treats remote as authoritative on conflict, which contradicts the V2 thesis (`local as source of truth`)

Net effect: the V2 promise — "your folder is the project; the cloud mirrors it" — only holds for filestore. KBs still feel like V1 to the user, and any work the user does locally on a KB can be silently overwritten by a remote pull.

The deferred items called out in Phase 1's review (`lastSyncAt` wiring, V1 → Phase 1 folder migration UX, first-bind confirm dialog, `openit-docs-<orgId>` cloud-side rename) are real but cosmetic compared to bringing a second engine onto the local-first model.

## Desired Outcome

KBs behave like filestore does after Phase 1:

- The bound `~/OpenIT/local/` folder is the source of truth for KB content.
- Discovery on the cloud is filtered to `openit-*` KBs only; unrelated user KBs are invisible to OpenIT.
- Every `openit-<name>` KB on the cloud gets its own `knowledge-bases/<name>/` folder locally, with its own manifest entry. New ones in either direction round-trip cleanly.
- The shared manifest architecture flaw (one collection's `collection_id` clobbering another's) is fixed for KBs the same way it was fixed for filestore — nested `{ [collectionId]: KbStatePersisted }` keyed by `collectionId`.
- Conflicts resolve local-first: local edits win on the `localChanged && remoteChanged` branch, with the existing `.server.<ext>` shadow-file mechanism preserved as the escape hatch.
- The sync engine's "tracked but missing on disk" re-fetch (added in Phase 1) applies to KB files too.

A user with `openit-runbooks` and `openit-faqs` KBs on the cloud should be able to `cd ~/OpenIT/local/knowledge-bases/runbooks/`, edit a markdown file, and have it propagate up within the next poll. Conversely, dropping a folder into `knowledge-bases/onboarding/` locally should create `openit-onboarding` on the cloud.

## Scope

### In

1. **KB engine local-first rewrite** — mirrors the Phase 1 filestore changes:
   - `openit-*` prefix filter on KB discovery
   - Nested per-collection manifest (`.openit/kb-state.json` → `{ [collectionId]: KbStatePersisted }`)
   - Per-collection routing (`openit-<name>` ↔ `knowledge-bases/<name>/`)
2. **Extract a shared `nestedManifest.ts`.** `filestoreManifest.ts` (134 LOC) already keys on `collectionId` and types its value as `KbStatePersisted` — the on-disk shape is engine-agnostic. Pull it into `src/lib/nestedManifest.ts`, parameterize the manifest filename (`fs-state.json` vs `kb-state.json`), and have both engines depend on it. Avoids a near-duplicate file and locks the shape in once.
3. **`lastSyncAt` wiring** — call `project_update_last_sync_at` from the reconciler success callback for both filestore and KB. Filestore was deferred in Phase 1's review (Finding #1); pick up here since we're already touching reconciler callbacks for KB. Cost is ~1 line per engine; benefit is unblocking any future "Last synced X ago" UI.
4. **Integration tests** — extend the existing `integration_tests/` framework to cover KB scenarios (`adapter-routing`, `filestore-sync` analogs), against real backend.

### Out (deferred to later phases)

- **Datastore / workflow / agent local-first sync** — different sync models (datastore = JSON rows, workflow = yaml, agent = json). Each warrants its own phase.
- **Push-in-poll-loop wiring.** The 60s poll already runs through `syncEngine.pullEntity`, which is bidirectional — local edits propagate up on the same tick. `pushAllToKb` / `pushAllToFilestore` stay as the manual "force-push everything" sweep wired to the Sync-tab commit handler. Revisit only if integration testing surfaces a gap.
- **Plugin overlay revision-gating** — orthogonal concern, own phase.
- **First-bind confirm dialog / "switch cloud orgs" UX** — UI work, separate ticket.
- **V1 → Phase 1 folder migration UX** — separate ticket; surface area is "what to do when `lastRepo` points at an orphaned `~/OpenIT/<oldOrgId>/`".
- **`openit-docs-<orgId>` cloud-side rename** — Phase 1 left these orphaned on remote; Phase 2 still doesn't migrate them.
- **Plan-limit pre-flight (SC-6)** — separate ticket.
- **Multi-collection custom-named filestore beyond `library` / `attachments`** — Phase 1's discovery flow already supports any `openit-<name>` collection; no extra Phase 2 work needed unless gaps surface during integration testing.
- **Generalizing to a `localFirstAdapter` abstraction across all engines** — defer until 3+ engines are running locally so the right shape is visible. The shared `nestedManifest.ts` extraction in this phase is deliberately narrow (just the on-disk manifest shape).

## Success Criteria

- [ ] When a user with creds connects and the cloud has `openit-runbooks` and `openit-faqs` KBs, both pull down to `knowledge-bases/runbooks/` and `knowledge-bases/faqs/` respectively
- [ ] When a user creates `knowledge-bases/onboarding/` locally with markdown files inside, on next sync `openit-onboarding` exists on the cloud with those files
- [ ] When the user has unrelated non-OpenIT KBs on the cloud (e.g. `customer-knowledge`), they are not pulled, not modified, and not visible to the engine
- [ ] When a user edits a KB file locally and remote has not changed, the local edit propagates up; when both changed, a `.server.<ext>` shadow file is written and a conflict event fires (existing Phase 1 mechanism)
- [ ] `.openit/kb-state.json` is nested per `collectionId`; an existing flat-format manifest from V1 is migrated forward on first read (same pattern as `filestoreManifest.ts`)
- [ ] Files tracked in the manifest but missing on disk are re-fetched from remote (no silent data loss)
- [ ] `cloud.json.lastSyncAt` updates after each successful filestore OR KB pull (Phase 1 deferral resolved)
- [ ] Integration tests cover: discovery filter, multi-KB routing, push from local-only, conflict shadow-file write, manifest migration from V1 flat format
- [ ] Existing filestore behaviour from Phase 1 is not regressed — `npm run test:integration` against filestore still passes
- [ ] `cargo test`, `npx vitest run`, `npx tsc --noEmit` all clean

## Resolutions (recorded 2026-04-29)

1. **Engine choice:** KB only. Datastore / workflow / agent stay deferred.
2. **`lastSyncAt` wiring:** in scope. Wire from both filestore and KB reconciler-success callbacks.
3. **Push-in-poll-loop:** out of scope. The bidirectional `syncEngine.pullEntity` already propagates local edits on the 60s tick; the `pushAll*` functions are intentional manual sweeps. Keep KB's shape matching filestore's.
4. **Shared `nestedManifest.ts`:** extract now. `filestoreManifest.ts` is 134 LOC and its value type is already `KbStatePersisted` — the shape is engine-agnostic.
5. **Linear ticket:** new sibling of PIN-5775, to be created.
6. **V1 KB orphans:** non-issue — V2 hasn't launched yet.
