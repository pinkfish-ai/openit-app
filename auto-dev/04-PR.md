File the PR.
Run the Cursor BugBot review loop.

## Triggering BugBot

- Add a comment on the PR with `@cursor review` on its own line. **The trigger phrase is exactly `@cursor review` — verbs like "please re-review", "take another look", "review again" do NOT fire a review.** The mention without that exact verb is a no-op (which surprised me in iter 7 of this PR).
- **Focus the review by listing specific concerns** below the trigger line. BugBot reads the comment body and prioritizes whatever you flag. Big payoff when you've made a non-obvious design choice you want vetted, or when the diff is broad and a generic review would scatter.

Example focused trigger:

```
@cursor review

Latest 5 commits since 15:36 — substantial batch.

Specifically interested in:
- The Rust `#[serde(default, skip_serializing_if = "Option::is_none")]`
  on the new field — backwards-compat with existing manifests?
- Force-push semantics in sync-resolve-conflict.mjs (`pulled_at_mtime_ms = 1`
  sentinel) — any race with concurrent local edits?
- `pushInFlightByRepo` Set vs a per-(repo, prefix) lock model — overkill
  or right-sized?
```

Note: the literal `@cursor review` line stands alone. Any focusing context goes on subsequent lines. Numbered or bulleted concerns get separately addressed in the response. Naming commits by SHA (or a date range) tells BugBot what scope to look at — without this it tends to re-scan the whole PR.

## Iterating

- Poll the PR every minute or so for the review.
- BugBot grades severity: **Critical / High / Medium / Low**. Each finding lands as an inline review comment with severity in the title.
- For each finding, either fix it or reply with a why-not-fix rationale on the thread itself.
- After fixing, resolve the thread. The `gh` CLI doesn't expose this — use the GraphQL mutation:

  ```bash
  gh api graphql -f query='mutation { resolveReviewThread(input: { threadId: "PRRT_…" }) { thread { isResolved } } }'
  ```

  Get thread IDs from:

  ```bash
  gh api graphql -f query='{ repository(owner: "ORG", name: "REPO") {
      pullRequest(number: N) { reviewThreads(last: 30) { nodes { id isResolved path line } } } } }'
  ```

- After fixing a batch, retrigger with `@cursor review` and **say what changed since the last review** — list the new commits and what they touched. Otherwise BugBot may rescan ground it already covered.

## Stop condition

Keep iterating until the only remaining findings are **Low-severity**. At that point reply on each Low thread with a rationale (or fix), resolve, and merge.

Watch for:
- **BugBot auto-pushes** — it can land a commit itself for trivial mechanical fixes (variable shadowing, dead-code, etc.). **Always `git pull` before iterating** or you'll silently revert its fix on your next push / rebase.
- **Pre-existing CI red on main** — known state for this repo (auto-generated firebase-helpers + cargo fmt). BugBot won't flag it, but if your PR's CI fails for the same reasons, that's not a regression.
- **Real issues vs. style nits** — BugBot is good at structural correctness and edge cases; less good at project-specific conventions. Push back on style nits that conflict with patterns in CLAUDE.md.
